### DTO、VO、POJO

#### DTO (Data Transfer Object)

**作用：在不同服务 / 接口 / 层之间传输数据**（比如前端→后端、服务 A→服务 B）。

核心场景

-   接收**前端传过来的参数**
-   微服务之间调用时传递数据
-   只包含**需要传输的字段**，不暴露敏感信息

```java
// 只接收前端传的账号和密码，不包含数据库字段
public class UserLoginDTO {
    private String username;
    private String password;
    // getter/setter
}
```



#### VO (View Object)

**作用：给前端页面展示数据**（Controller → 前端）。

核心场景

-   封装**返回给前端的最终数据**
-   可以组合多个表的数据
-   只保留页面需要的字段

```java
// 前端只需要展示这些，不需要密码、创建时间等
public class UserInfoVO {
    private Long userId;
    private String username;
    private String avatar;
    private Integer age;
    // getter/setter
}
```



#### POJO (Plain Old Java Object)

一种**统称、父概念**，不是具体的业务对象。

特点

-   只有**私有属性 + getter/setter**
-   无继承、无复杂逻辑
-   不依赖任何框架
-   **DTO、VO 本质上都是 POJO**

```java
// 标准POJO
public class User {
    private Long id;
    private String name;

    // getter、setter
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
}
```

### 后端接收前端参数的格式

在苍穹外卖项目中，后端中的 Controller 需要接收来自前端的请求。

前端的常见 HTTP 请求为：

*   GET：查询、获取数据
*   POST：新增、提交数据
*   PUT：修改、更新数据
*   DELETE：删除数据

对应的三种传参方式：

*   路径参数
*   Query 参数
*   JSON 参数

对应这四种请求和三种传参方式，后端接收参数的方式概括为如下表格：

| 请求方式   | 主要用途 | 前端常用传参格式 | 后端接收注解             | 苍穹外卖常见场景 |
| ---------- | -------- | ---------------- | ------------------------ | ---------------- |
| **GET**    | 查询数据 | 路径 `/user/100` | **@PathVariable**        | 根据 ID 查询     |
|            |          | Query `?name=xx` | **省略 / @RequestParam** | 分页、条件查询   |
| **POST**   | 新增数据 | JSON             | **@RequestBody**         | 新增员工、登录   |
| **PUT**    | 修改数据 | JSON             | **@RequestBody**         | 修改员工、起停售 |
| **DELETE** | 删除数据 | 路径 `/user/100` | **@PathVariable**        | 根据 ID 删除     |

具体代码实现：

（1）路径参数

形式：

URL：`/admin/employee/100`

参数直接写在 URL 路径里面，用 `{}` 占位，接收注解为 `@PathVariable`

```java
@GetMapping("/admin/employee/{id}")
public Result getById(@PathVariable Long id)
```

（2）Query 参数

形式：

URL：`/admin/employee/page?name=张三&page=1&pagesize=10`

URL 后面用 `?` 开始，GET、POST 请求都能用，接收注解为 `@RequestParam`（可不写，默认就是）

```java
@GetMapping("/admin/employee/page")  // 如果已经在类注解中已经写明 @RequestMapping("/admin/employee")，可以省略重复的部分
public Result page(String name, Integer page, Integer pageSize)
```

当然，也可以将 `String name`、`Integer page`、`Integer pageSize` 这三项封装为一个DTO，然后直接用这个 DTO 来接收参数：

```java
@GetMapping("/admin/employee/page")
public Result page(EmployeePageQueryDTO employeePageQueryDTO)
```

其中 `EmployeePageQueryDTO` 的定义为：

```java
package com.sky.dto;

import lombok.Data;
import java.io.Serializable;

@Data
public class EmployeePageQueryDTO implements Serializable {
    // 员工姓名
    private String name;
    // 页码
    private int page;
    // 每页显示记录数
    private int pageSize;
}
```

（3）JSON 参数（请求体参数）

主要用于 POST、PUT 请求，用于传递复杂参数，接收注解为：`@RequestBody`

```java
@PostMapping("/admin/employee")
public Result save(@RequestBody EmployeeDTO employeeDTO)
```

## 三层架构

在三层架构中，`Mapper` 就是 `Dao` 的现代实现形式，传统 `Dao` 接口在 MyBatis 持久层框架下，统一改名叫 `Mapper`，二者职责完全一致，只是叫法、编码形态不同，**三层架构本质没变**。

### Controller / Service / Dao

标准三层架构划分（**业务分层，职责解耦**），自上到下：

1.  **Controller 控制层（表现层）**

    接收前端请求、参数校验、调用 Service、封装返回结果、处理响应。

    对应苍穹外卖：`controller` 包下所有类。

2.  **Service 业务层**

    编写**核心业务逻辑**、事务控制、调用持久层、业务规则判断。

    对应苍穹外卖：`service / service.impl` 包。

3.  **Dao 持久层（数据访问层）**

    只负责**和数据库交互**：增删改查，不写业务逻辑。

    传统写法：`Dao 接口 + DaoImpl 实现类`。

>   调用链路：
>
>   **前端请求 → Controller → Service → Dao → MySQL**

### 传统 Dao 长什么样

在没有 MyBatis、MyBatis-Plus 之前，纯 JDBC / JdbcTemplate 时代，严格叫 **Dao**。

1、传统 Dao 接口

```java
public interface UserDao {
    User getUserById(Long id);
    int insertUser(User user);
}
```

2、Dao 实现类（DaoImpl）

里面写原生 JDBC 代码：获取连接、写 SQL、执行、封装结果、关闭连接。

```java
public class UserDaoImpl implements UserDao {
    @Override
    public User getUserById(Long id) {
        // 1. 获取数据库连接
        // 2. 编写 SQL
        // 3. 创建 Statement/PreparedStatement
        // 4. 执行查询、封装 User 对象
        // 5. 关闭资源
        return user;
    }
}
```

**痛点**：

-   大量重复代码（连接、关闭资源、参数设置）
-   SQL 和 Java 代码硬耦合，改 SQL 要改 Java 文件
-   开发效率极低

于是出现了**持久层框架 MyBatis**，并把 `Dao` 重新命名为 `Mapper`。

### Controller / Service / Mapper

#### 1、框架约定

MyBatis 官方统一将数据访问层接口命名为 Mapper，这是行业通用约定：

-   传统 JDBC $\to$ 叫 `Dao`
-   MyBatis / MyBatis-Plus $\to$ 统一叫 `Mapper`

`Mapper` = 新版 `Dao`，职责 100% 等价：只做数据库 CRUD。

#### 2、MyBatis 下 Mapper 的形态

MyBatis 只需要接口，不需要手写实现类，框架自动动态生成代理对象完成数据库操作。

（1）Mapper 接口（对应原来的 Dao 接口）

```java
// 苍穹外卖里典型的 Mapper
@Mapper
public interface UserMapper {
    // 方法名、参数、返回值 和 SQL 映射
    User getUserById(Long id);
    int insertUser(User user);
}
```

（2）XML / 注解 SQL（替代原来 DaoImpl 里的 JDBC 代码）

SQL 单独抽离到 `mapper/*.xml` 文件或注解中，解耦 Java 与 SQL。

调用链路完全没变：

>   Controller → Service → Mapper (原 Dao) → 数据库

#### 3、包名对比

| 分层       | 传统 JDBC 项目 | 苍穹外卖 (MyBatis)     | 职责               |
| ---------- | -------------- | ---------------------- | ------------------ |
| 控制层     | controller     | controller             | 接收请求、响应前端 |
| 业务层     | service        | service                | 业务逻辑、事务     |
| 持久层接口 | dao            | mapper                 | 定义数据库操作方法 |
| 持久层实现 | dao.impl       | 无（MyBatis 动态代理） | 框架自动实现       |

#### 4、三层架构 + Mapper 的完整执行流程

以「根据 ID 查询用户」为例：

1.  `Controller`：接收前端传的 `id`，调用 `userService.getUserById(id)`
2.  `Service`：接收参数，做业务判断，调用 `userMapper.selectById(id)`（可加事务）
3.  `Mapper`：MyBatis 找到对应 SQL，执行查询，返回数据
4.  数据原路返回：Mapper → Service → Controller → 前端

**原则始终不变**：

-   Controller 不碰数据库
-   Service 只写业务，不写 SQL



## Codex 总结

### 1. 基本类型 vs 包装类型

基本类型是 `int`、`long`、`double`、`boolean` 这种，特点是轻量、不是对象、不能为 `null`。
包装类是 `Integer`、`Long`、`Double`、`Boolean`，本质是对象，可以为 `null`。

为什么项目里更常用包装类？

-   数据库字段查出来后，可能为空，要允许 `null`
-   泛型只能放对象，不能放基本类型，比如 `List<Long>` 可以，`List<long>` 不行
-   很多框架参数绑定、JSON 转换、MyBatis 映射都更常和包装类打交道

:warning: 基本类型不是对象，包装类是对象。

:warning: 包装类可以为 `null`，所以更适合数据库和框架。

:warning: 泛型里只能放包装类，不能放基本类型。



### 2. 自动装箱和拆箱

自动装箱：long -> Long
自动拆箱：Long -> long

比如：

```java
Long userId = 1L;     // 自动装箱
long id = userId;     // 自动拆箱
```

这东西平时很方便，但有一个高频坑：
如果包装类是 null，拆箱时会空指针。

比如：

```java
Long userId = null; 
long id = userId; // NullPointerException
```

这和你的项目很相关。比如统计代码里：

```java
Integer totalOrderCount = orderMapper.countByMap(map);
```

如果你后面直接拿它当基本类型算，而且没判空，就容易出问题。

因此，应该需要“先判空再处理“。

:warning: 自动拆箱很方便，但 `null` 拆箱会空指针。



### 3. `String` 是什么，为什么这么常用

`String` 是不可变对象。
意思是：字符串一旦创建，内容不能改，所谓“修改”其实是新建了一个字符串。

项目里最常见的字符串用途：

-   token
-   订单号
-   地址拼接
-   Redis key
-   日志输出
-   报表里的日期串



### 4. `StringBuilder` 什么时候用

如果只是偶尔拼一次字符串，直接 `+` 可以。如果在循环里频繁拼接，优先 `StringBuilder`。

原因很简单：
`String` 不可变，每次 `+` 都可能创建新对象；循环里一直拼，会多创建很多中间字符串。

在 `OrderServiceImpl.java` 的 `getOrderDishesStr()` 里，你现在是先转成 `List<String>`，再 `String.join("", orderDishList)`。这也是一种减少手写拼接混乱的办法：

```java
    /**
     * 根据订单 id 获取菜品信息字符串
     */
    private String getOrderDishesStr(Orders orders) {
        // 查询订单菜品详情（订单中的菜品及其数量）
        List<OrderDetail> orderDetailList = orderDetailMapper.getByOrderId(orders.getId());

        // 将每一条订单菜品信息拼接为字符串（格式：宫保鸡丁*3;鱼香肉丝*2）
        List<String> orderDishList = orderDetailList.stream().map(x -> {
            String orderDish = x.getName() + "*" + x.getNumber() + ";";
            return orderDish;
        }).collect(Collectors.toList());

        // 将该订单对应的所有菜品信息拼接在一起
        return String.join("", orderDishList);
    }
```

如果以后手写循环拼菜品描述，`StringBuilder` 会更合适：

```java
    /**
     * 根据订单 id 获取菜品信息字符串
     */
    private String getOrderDishesStr(Orders orders) {
        // 查询订单菜品详情（订单中的菜品及其数量）
        List<OrderDetail> orderDetailList = orderDetailMapper.getByOrderId(orders.getId());

        StringBuilder sb = new StringBuilder();
        for (OrderDetail orderDetail : orderDetailList) {
            sb.append(orderDetail.getName()).append("*").append(orderDetail.getNumber()).append(";");
        }
        
        return sb.toString();
    }
```

:warning: 循环拼接字符串优先考虑 `StringBuilder`。



### 5.  什么是类，什么是对象

最简单理解：

-   `类` 是模板
-   `对象` 是根据模板创建出来的具体实例

比如你项目里的 `Orders.java`：

-   `Orders` 这个类，是“订单”这种数据结构的模板
-   `new Orders()` 创建出来的，才是某一个具体订单对象

你在 OrderServiceImpl.java 里经常写：

```
Orders orders = new Orders();
```

这就是“根据类创建对象”。

一句话记忆：
**类定义结构，对象承载数据。**



### 6. 为什么要封装

封装就是：
**不让外部直接乱碰属性，而是通过方法来访问和修改。**

Java 里最常见的方式是：

-   属性用 `private`
-   提供 `getter/setter`

比如：

```java
private Long id;
public Long getId() {
    return id;
}
public void setId(Long id) {
    this.id = id;
}
```

为什么这么做？

-   可以控制谁能改数据
-   可以在设置值时做校验
-   更符合面向对象设计

你项目里 `Entity`、`DTO`、`VO` 大多都是这种风格，所以本质上都是在遵守 Java 的封装习惯。

注意：
虽然很多时候 setter 看起来只是“赋值”，但它仍然是一层边界。以后如果要加校验，比 public 字段更容易扩展。



### 7.  构造方法是什么

构造方法是在创建对象时执行的方法，用来初始化对象。

最常见两种：

**无参构造**

```java
Orders orders = new Orders();
```

**有参构造**

```java
User user = new User(id, name);
```

为什么你项目里大量依赖无参构造？

-   `Spring`
-   `MyBatis`
-   `Jackson`
-   `BeanUtils`

这些框架经常需要先“new 一个空对象”，再把属性一个个塞进去。
所以很多数据对象都很适合保留无参构造。

一句话记忆：
**框架喜欢无参构造，因为它们要先创建对象，再做属性填充。**



### 8. 什么是 JavaBean

你项目里最常见的对象，其实大多都属于 JavaBean 风格。

通常满足这些特点：

-   有私有属性
-   有无参构造
-   有 getter/setter
-   用来承载数据

比如：

-   `Employee`
-   `Orders`
-   `DishDTO`
-   `OrderVO`

为什么 JavaBean 在后端项目里这么常见？

-   框架兼容性好
-   适合做参数绑定
-   适合做数据库映射
-   适合做 JSON 序列化和反序列化

所以你可以把 `Entity/DTO/VO` 理解成：
**不同职责的 JavaBean。**

职责区别再复习一遍：

-   `Entity`：更贴近数据库表
-   `DTO`：接收前端参数
-   `VO`：返回给前端的数据

>   如果一个类只有全参构造，**没有无参构造**，可能会对哪些框架场景造成不便？
>
>   答：**没有无参构造，会让很多依赖反射和属性填充的框架使用起来不顺手。**比如：
>
>   -   MyBatis 查询结果封装对象
>   -   Jackson 反序列化请求体
>   -   BeanUtils、反射工具创建对象
>   -   一些框架先实例化空对象再逐个填充属性的场景

>   你项目里很多地方用了 BeanUtils.copyProperties，它为什么更适合在 Entity/DTO/VO 这种结构相近的对象之间使用？
>
>   答：`BeanUtils.copyProperties` 适合在字段名和字段类型大体一致的 JavaBean 之间做快速拷贝，比如 `DTO -> Entity`、`Entity -> VO`。如果结构差异很大，或者需要复杂转换，它就不够合适了。



### 9. Lombok 在帮你做什么

Lombok 不是新语法，它只是帮你少写样板代码。

你项目里常见的：

-   `@Data`
-   `@Builder`
-   `@NoArgsConstructor`
-   `@AllArgsConstructor`

它们一般会帮你生成：

-   `getter/setter`
-   `toString`
-   `equals/hashCode`
-   构造方法
-   builder 模式代码

比如你在项目里经常看到：

```java
Orders orders = Orders.builder().
    id(ordersDB.getId()).
    status(Orders.COMPLETED)
    .build();
```

这就是 `@Builder` 带来的好处。相比一堆 `setter`，它的优点是：

-   可读性更强
-   更适合只设置部分字段
-   不容易漏字段或写错顺序

但也别神化它，本质还是在创建对象。



### 10. Java 异常大体怎么分

你现在先记最实用的一层：

-   Exception
-   RuntimeException

面试和项目里最常用的区分就是：

*   **checked exception**
    编译器强制你处理
    比如某些 IO、反射、外部调用异常

*   **unchecked exception**
    也就是运行时异常，继承 RuntimeException
    编译器不强制你处理

你项目里的业务异常，基本都属于第二类。比如：

-   `OrderBusinessException`
-   `DeletionNotAllowedException`
-   `LoginFailedException`

这些异常本质是：
**业务不满足条件，所以要中断流程并返回错误信息。**

>   -   业务异常：规则不满足，比如购物车为空、订单状态不对、超出配送范围
>   -   系统异常：程序或环境出了问题，比如数据库连不上、空指针、网络调用崩了



### 11. 为什么业务异常常继承运行时异常

这是后端项目里很高频的设计。

原因主要有 3 个：

1.  业务校验失败本来就是运行阶段才知道的
    比如“购物车是不是空”“订单状态对不对”，编译时不可能知道。
2.  不想在每一层都写很多 `throws` 和 `try-catch`
    如果业务异常都设计成受检异常，controller、service 一层层传会很重。
3.  更适合和 Spring 事务、全局异常处理配合
    抛出去后让统一异常处理器接住，代码更干净。

你项目里这个设计就很典型：

-   service 里发现业务不满足条件
-   直接 `throw new xxxBusinessException(...)`
-   最后由全局异常处理器统一转换成前端能看懂的错误结果



### 12. 你项目里的异常链路

你可以直接看：

-   `GlobalExceptionHandler.java`
-   `OrderServiceImpl.java`
-   `DishServiceImpl.java`

链路大概是这样：

1.  controller 收到请求
2.  service 做业务判断
3.  如果条件不满足，抛业务异常
4.  `GlobalExceptionHandler` 统一捕获
5.  返回统一格式的 `Result.error(...)`

这套好处非常大：

-   controller 更干净
-   错误处理风格统一
-   前端更容易接
-   业务语义更清晰



### 13. 为什么不建议到处写 try-catch

很多初学者喜欢这样写：

```java
try {
    // 业务逻辑
} catch (Exception e) {
    return Result.error("出错了");
}
```

问题是：

-   错误信息被抹平了
-   controller/service 会变得很乱
-   真正的业务异常和系统异常混在一起
-   后续日志、排查、统一处理都不方便

所以你项目现在这种：

-   业务层主动抛业务异常
-   全局统一接住
    是更典型的后端写法。



### 14. 全局异常处理器在做什么

`GlobalExceptionHandler` 本质上就是：

**把分散在各处抛出的异常，统一收口成接口响应。**

你项目里做了两类处理：

-   处理业务异常 `BaseException`
-   处理 SQL 唯一约束异常 `SQLIntegrityConstraintViolationException`

这说明你已经有“分类处理异常”的意识了，这点挺好。



### 15. 为什么要有泛型

你项目里经常有这种代码：

```java
List<OrderDetail> orderDetailList = orderDetailMapper.getByOrderId(id);
List<DishVO> list = dishService.listWithFlavor(dish); Map map = new HashMap();
```

其中 `List<OrderDetail>` 里的 `<OrderDetail>` 就是泛型。

泛型的作用很直接：

-   限制容器里放什么类型
-   让编译器提前帮你检查类型错误
-   取数据时少写强转，代码更安全

比如：

```java
List<String> list = new ArrayList<>();
list.add("abc");
// list.add(123); // 编译期就报错
```

如果没有泛型，很多错误会拖到运行时才暴露。

你现在先记住：
**泛型的核心价值就是“类型安全”。**



### 16. 数组 vs `ArrayList`

**数组**

-   长度固定
-   可以存基本类型，也可以存对象
-   适合已知长度、简单场景
-   功能比较原始，没有 `add/remove` 这些便捷方法

**`ArrayList`**

-   长度可动态变化
-   只能放对象，不能直接放基本类型
-   提供很多常用方法，比如 `add/get/remove/size`
-   更适合项目开发中的“查出一批数据再处理”



### 17. 为什么项目里经常要拷贝对象

你项目里最常见的对象有三类：

-   `DTO`：接前端参数
-   `Entity`：贴近数据库表
-   `VO`：返回给前端

它们经常字段很像，但职责不同，所以不能直接混用。
这时候就需要“拷贝公共字段”。

你项目里很典型的地方：

-   `OrderServiceImpl.java`
-   `DishServiceImpl.java`
-   `SetmealServiceImpl.java`

比如你经常会看到：

```java
Orders orders = new Orders();
BeanUtils.copyProperties(ordersSubmitDTO, orders);
```

意思不是“这两个对象完全一样”，而是：
**把名字相同、类型兼容的属性，从 DTO 拷给 Entity。**



### 18. BeanUtils.copyProperties 到底在做什么

你先把它理解成一句话：

**按照属性名，把源对象里的值复制到目标对象。**

比如：

```java
Dish dish = new Dish();
BeanUtils.copyProperties(dishDTO, dish);
```

这通常适用于：

-   `DTO` -> `Entity`
-   `Entity` -> `VO`
-   一个结构相近的对象 -> 另一个结构相近的对象

它的优点很明显：

-   少写很多 setter
-   代码更短
-   结构相近时开发效率很高

所以在教学项目和常规业务项目里，它非常常见。

>   “BeanUtils.copyProperties 适合结构相近的 JavaBean 做快速字段拷贝，复杂映射场景应该手动处理，避免隐式错误。像 id、状态、时间、集合字段这些属性，拷贝时要特别注意语义和副作用。”

### 20. 接口和实现类

-   接口：定义方法规范
-   实现类：真正写业务逻辑

```java
public interface OrderService {
    OrderSubmitVO submitOrder(OrdersSubmitDTO ordersSubmitDTO);
}
```

```java
@Service
public class OrderServiceImpl implements OrderService {
    public OrderSubmitVO submitOrder(OrdersSubmitDTO ordersSubmitDTO) {
        // 真正的下单逻辑
    }
}
```

1.  接口定义规范，实现类负责落地。
2.  保留接口不只是为了多个实现类，更是为了抽象边界。
3.  controller 不写复杂业务，service 才是业务核心层。
4.  mapper 只负责数据访问，不负责业务编排。
5.  面向接口编程的本质是“依赖抽象，降低耦合”。
6.  事务通常放在 service 层，因为它要包住完整业务链路。



### 21. 什么是注解

你可以先把注解理解成：

**给类、方法、字段加上的一种“说明标签”。**

比如你项目里有很多注解：

-   `@RestController`
-   `@Service`
-   `@Autowired`
-   `@Transactional`
-   `@ApiOperation`
-   `@AutoFill`

它们本身不是业务逻辑，更多是在表达：

-   这个类是什么角色
-   这个方法有什么特征
-   这个字段要怎么处理

最关键的一点：
**注解本身不会自动完成复杂逻辑，真正干活的是读取注解的框架或代码。**

比如：

-   `@Service` 被 Spring 读取，然后注册成 Bean
-   `@Transactional` 被 Spring AOP 读取，然后织入事务逻辑
-   `@AutoFill` 被你的 AutoFillAspect 读取，然后执行自动填充



1.  注解本质上是一种标记。
2.  注解本身不干活，真正干活的是读取注解的代码。
3.  反射可以在运行时获取类、方法、注解并执行方法。
4.  `@AutoFill` 是你项目里的自定义注解，用来标记公共字段自动填充。
5.  反射适合做通用能力，但不适合在普通业务里到处滥用。



“`@AutoFill` 适合放在切面里，因为它是典型的横切逻辑，统一处理能减少重复代码并保证规则一致。切面里通过反射去查找 setter 并执行，这种方式很灵活，但如果方法名写错，通常要到运行时才会暴露问题。这也说明反射更适合封装框架层或通用能力，而不适合在普通业务代码里到处使用。”
